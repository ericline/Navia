## Key Decisions & Reasoning

| Decision | Reasoning | Alternatives Considered |
|----------|-----------|------------------------|
| Procedural shape generation instead of fixed templates | Every trip gets a truly unique constellation rather than one of 6 templates with jitter. Seeded PRNG ensures deterministic shapes per trip ID. | More templates (12-15), random placement with force-directed layout |
| Straight lines instead of bezier curves | Real constellations use straight connections between stars. Sharp angles make the patterns more recognizable and distinctive. | Polylines with rounded joints, slight curve at vertices |
| No backend changes for constellation storage | Constellation shape is derived deterministically from `tripId + dayCount`, so it doesn't need to be stored. Same inputs always produce the same shape. | Store constellation points in DB, allow user customization |
| Separate `TripConstellation` vs reusing `ConstellationPath` | `ConstellationPath` is week-sliced for the itinerary view; `TripConstellation` renders the full trip and supports reveal animation + compact mode. Different enough use cases to warrant separate components. | Single component with many mode flags |
| Lazy-load days in `PastTripCard` | Past trips may have many days/activities; loading on mount keeps the home page snappy while still showing constellations. | Preload all trip days on home page load |
